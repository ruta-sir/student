## Project Overview
 
Wayfinding World is a Canvas RPG learning platform for the CS Pathway. My work spans three trimesters and two backend stacks.
 
**Tri 1–2 — Microblog Groups:** Full-stack group posting system. Users post as a group or individual. Involved extending the DB schema, building Flask endpoints, integrating with the Spring groups API via JWT, and building a post composer with context-aware routing.
 
**Tri 3 — Skill Passport:** Passport-themed full-screen modal that fetches a student's skill snapshot history from a Python backend. Shows current levels (Coding, Collaboration, Problem Solving, Initiative) as progress bars and logs historical snapshots as stamped passport entries. Built as a standalone `SkillPassport` class — handles its own fetching, rendering, and DOM cleanup.
 
The broader platform: Identity Forge (persona selection), personality quiz, Code Hub (HTML learning), Sprint Success (agile missions), NPC-guided navigation.

## Personal & Social Relevance
 
**Problem 1 — Group posting without identity loss.** Students wanted to post under a group name without losing their individual profile. I added an `author_type` field that switches attribution between user and group, with permission checks blocking unauthorized group posts.
 
**Problem 2 — Invisible skill growth.** Students finish projects but rarely see their own progression. The Skill Passport turns raw backend data into a visual record of growth — framed as a journey, not a grade. My mom does similar data aggregation work professionally, which pushed me to build this more seriously.
 
**Ethical considerations:**
- **Privacy** — all data gated behind JWT cookie auth; nothing readable without a valid session
- **Security** — group post endpoints enforce 401 → 403 → 404 permission chain
- **Equity** — skill scores use a 1–6 scale with progress bars, not letter grades
- **Accessibility** — Skill Passport uses high-contrast colors (dark green, gold) and is keyboard-closeable; NPC dialogue requires no prior game experience

## Data Structures
 
**1. Graph — Adjacency List**
Hub zones are nodes; transitions between them are edges. Adjacency list is memory-efficient for sparse graphs where most zones connect to only 2–3 others.
 
```java
Map<Integer, List<Edge>> adjacencyList = new HashMap<>();
 
public void addEdge(int from, int to, double weight) {
    adjacencyList.computeIfAbsent(from, k -> new ArrayList<>())
                 .add(new Edge(to, weight));
}
// O(1) amortized insertion; O(degree) neighbor traversal
```
 
**2. HashMap**
Used in `PersonaService` for O(1) persona lookup and in Microblog Groups for UID-to-person-ID resolution across the Flask–Spring boundary.
 
```java
HashMap<String, Persona> personaMap = new HashMap<>();
personaMap.put("achiever", new Persona("achiever", "🏆", "Achievement"));
Persona selected = personaMap.get(studentChoice); // O(1)
```
 
**3. ArrayList**
BFS path results, quiz question order, and Skill Passport snapshot history all use ArrayList for dynamic sizing and index-based reverse-order rendering.
 
```java
List<Location> path = new ArrayList<>();
// filled by BFS reconstructPath()
Collections.sort(path, Comparator.comparing(Location::getName));
```
 
**4. Queue**
BFS traversal uses a `LinkedList` queue for FIFO node processing. Also used in the level injection pattern — sub-levels splice in at `currentLevelIndex + 1` and process in order.
 
```java
Queue<Integer> queue = new LinkedList<>();
queue.offer(startId);
while (!queue.isEmpty()) {
    int current = queue.poll();
    for (Edge e : adjacencyList.getOrDefault(current, new ArrayList<>())) {
        if (!visited.contains(e.getTo())) queue.offer(e.getTo());
    }
}
```
 
**5. Set**
`HashSet` tracks visited nodes in BFS (O(1) contains check) and enforces unique user roles — no duplicates by definition.
 
```java
Set<Integer> visited = new HashSet<>();
if (!visited.contains(neighbor.getTo())) {
    visited.add(neighbor.getTo());
    queue.offer(neighbor.getTo());
}
```

## Algorithms
 
**1. BFS Pathfinding**
Finds the shortest path between hub zones (e.g. Identity Forge → Code Hub). Chosen over Dijkstra's because transitions are unweighted — BFS guarantees shortest path with less overhead.
 
- Time: O(V + E) — each node and edge visited once
- Space: O(V) — queue and visited set
```java
public List<Location> findPath(int startId, int endId) {
    Queue<Integer> queue = new LinkedList<>();
    Map<Integer, Integer> cameFrom = new HashMap<>();
    Set<Integer> visited = new HashSet<>();
    queue.offer(startId);
    visited.add(startId);
    while (!queue.isEmpty()) {
        int current = queue.poll();
        if (current == endId) break;
        for (Edge e : adjacencyList.getOrDefault(current, new ArrayList<>())) {
            if (!visited.contains(e.getTo())) {
                visited.add(e.getTo());
                cameFrom.put(e.getTo(), current);
                queue.offer(e.getTo());
            }
        }
    }
    return reconstructPath(cameFrom, startId, endId);
}
```
 
**2. Sorting**
Persona compatibility ranking (Java `Comparator`) and Skill Passport reverse-chronological history (JS `Array.sort()`). Both O(n log n) — TimSort.
 
```java
candidates.sort(Comparator.comparingDouble(
    s -> compatibilityScore(targetPersona, s.getPersona())
).reversed());
```
 
```javascript
snapshots.sort((a, b) => new Date(b.createdAt) - new Date(a.createdAt));
```
 
**3. Search**
Location lookup by name and most-recent snapshot retrieval by userId. O(log n) with an indexed DB column.
 
```python
snapshots = SkillSnapshot.query.filter_by(user_id=user_id)\
                               .order_by(SkillSnapshot.created_at.desc()).all()
latest = snapshots[0] if snapshots else None
```
 
**4. Hashing**
BCrypt for password storage — salted + cost factor makes brute-force expensive. HashMap's internal hashing gives O(1) lookups for personas and UID resolution.
 
```java
String hashed = BCrypt.hashpw(rawPassword, BCrypt.gensalt());
boolean valid  = BCrypt.checkpw(rawPassword, hashed);
```

## Object-Oriented Design
 
**Abstraction**
`Waypoint` defines the contract for all navigable entities. The game loop calls `getType()` and `getDescription()` without knowing whether it's a `Location` or `Landmark`.
 
```java
public abstract class Waypoint {
    protected String name;
    protected double x, y;
    public abstract String getType();
    public abstract String getDescription();
}
```
 
`SkillPassport` does the same in JavaScript — all fetch/render/cleanup is hidden behind `open()` and `close()`.
 
**Encapsulation**
Private fields with validated setters. Score boundaries are enforced internally so callers can't corrupt the data model.
 
```java
public class SkillSnapshot {
    private double codingAbility; // 1–6 scale
    public void setCodingAbility(double score) {
        if (score < 1 || score > 6)
            throw new IllegalArgumentException("Score must be 1–6");
        this.codingAbility = score;
    }
}
```
 
**Inheritance**
`Landmark` extends `Location`, adding a `category` field for named points of interest (gates, NPC positions, zone entrances).
 
```java
public class Landmark extends Location {
    private String category; // "hub", "gate", "npc", "exit"
    @Override
    public String getDescription() {
        return "[" + category.toUpperCase() + "] " + super.getName();
    }
}
```
 
**Polymorphism**
The game loop iterates `List<Waypoint>` with mixed types. `getDescription()` dispatches to the correct subclass at runtime — no type-checking needed.
 
```java
List<Waypoint> waypoints = new ArrayList<>();
waypoints.add(new Landmark("Identity Forge", 100, 200, "hub"));
waypoints.add(new Location("Hallway Node A", 150, 200));
for (Waypoint w : waypoints) {
    System.out.println(w.getDescription()); // correct subclass, automatically
}
```
 
**Design Patterns**
 
| Pattern | Where | Why |
|---|---|---|
| MVC | Flask controllers / service / SQLite models | Separates routing, logic, data |
| Repository | `LocationRepository`, `SkillSnapshotRepository` | Consistent DB interface |
| Singleton | Flask app, DB connection | Prevents resource conflicts |
| Factory | `PersonaService.createPersona(type)` | Centralizes object construction |

## Help System
 
**How to use Wayfinding World:**
1. Log in with your student credentials
2. Enter **Identity Forge** — pick your persona archetype (Student, Social, Achievement, Fantasy)
3. Travel to **Wayfinding World** via the teleport gate that unlocks after selection
4. Complete the **Personality Quiz** — NPC gatekeepers guide each section; results go to the teacher dashboard
5. Visit **Code Hub** for HTML challenges or **Sprint Success** for agile missions — skills log to your Passport automatically
6. Open your **Skill Passport** any time to see current levels and full snapshot history
**For future engineers — setup:**
```bash
git clone https://github.com/blackstar3092/wayfinding_pages.git
cd wayfinding-world
pip install -r requirements.txt
python app.py
```
 
**Key files:**
- `skill_passport.js` — standalone SkillPassport class (fetch, modal, tabs, cleanup)
- `graph_service.py` / `GraphService.java` — BFS, adjacency list
- `persona_service.py` — persona assignment + compatibility scoring
- `microblog_controller.py` — group post creation + permission enforcement
- `levels.js` — level list; splice at `currentLevelIndex + 1` for sub-level injection
**Adding a new hub zone:** Insert a `Location` row (name, x, y, type) → add edges → add zone to `levels.js`.
 
**Adding a persona type:** Add to `personaMap` in `PersonaService` → add archetype card to Identity Forge canvas → map quiz answers to new type.
 
**SVG sprites:** Generated as inline data URIs — no image files. Each frame is a side-by-side `<g>` group. Edit in the JS sprite sheet generator directly.

## Retrospective
 
**What went well:**
- `SkillPassport` standalone class dropped cleanly into any game level with zero coupling
- Learning Flask *and* Spring in the same year gave real perspective on how REST API design differs across frameworks — and how to connect them
- Level injection splice pattern (`currentLevelIndex + 1`) made dynamic sub-levels clean and non-destructive
- Multi-stage permission checking (401 → 403 → 404) in Microblog Groups made the feature feel production-realistic
**What I'd do differently:**
- `try/catch` + `res.ok` on every fetch from day one — the silent 404 in Skill Passport cost real time
- Push local repo work incrementally, not all at once
- Plan the ERD earlier — `microblog_posts` needed mid-sprint refactoring when group requirements evolved
**What I learned:**
Three trimesters, two frameworks, one platform. The Skill Passport specifically pushed me to think beyond "does it work" — toward "can a student see themselves in it." That shift, from functional to meaningful, is what I'll take forward.
